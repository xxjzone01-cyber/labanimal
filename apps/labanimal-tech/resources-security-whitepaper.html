第一章：安全概述与设计理念

LabAnimal 将数据安全视为最高优先级。作为实验动物合规管理系统，我们处理的数据涉及研究协议、动物福利记录和审计追踪，这些数据的完整性和保密性至关重要。

我们的安全设计遵循以下核心原则：

  • 纵深防御：多层安全控制，不依赖单一安全机制
  • 最小权限：每个组件只拥有完成工作所需的最小权限
  • 零信任：所有请求都必须经过验证和授权
  • 透明开源：核心安全实现公开可审计

第二章：数据加密架构

2.1 传输加密

所有数据传输使用 TLS 1.3 加密，这是目前最安全的传输层协议版本。我们强制启用 HTTPS，所有 HTTP 请求都会自动重定向到 HTTPS。

Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

HSTS（HTTP Strict Transport Security）头确保浏览器始终使用 HTTPS 连接，防止降级攻击。

2.2 存储加密

密码使用 bcrypt 哈希存储，cost factor 为 12。这意味着即使数据库泄露，攻击者也无法在合理时间内破解密码。

密码哈希示例：
原始密码: MySecurePass123
bcrypt 哈希: $2b$12$LJ3m4ys3Lz0YBODCJn.MqOXPgFaHsVpJSKFTLKYqlVJTFHY.GGHKG

第三章：多租户数据隔离

LabAnimal 采用 PostgreSQL 行级安全（RLS）实现严格的多租户数据隔离。这是数据库层面的强制隔离，即使应用代码出现漏洞，也无法跨租户访问数据。

CREATE POLICY tenant_isolation ON animals
  USING (lab_id = current_setting('app.current_lab_id'));

每个数据库查询都会自动附加租户过滤条件，确保用户只能访问自己实验室的数据。这种隔离是强制性的，无法被绕过。

第四章：认证与授权机制

LabAnimal 使用 JWT（JSON Web Token）进行无状态认证。Token 使用 RS256 算法签名，这是一种非对称签名算法，比对称签名更安全。

4.1 密码策略

  • 最小 8 位字符
  • 必须包含大小写字母和数字
  • bcrypt 加密存储（cost factor 12）
  • 登录失败速率限制：5 次/分钟

4.2 角色权限

基于角色的访问控制（RBAC）定义了四种角色：Owner、Admin、Member、Viewer，每种角色拥有不同的权限集。

第五章：审计追踪与合规

为满足 21 CFR Part 11 合规要求，LabAnimal 实现了完整的审计追踪系统。每条审计记录包含操作类型、操作者、时间戳和数据快照。

审计日志采用 SHA-256 哈希链技术，每条记录的哈希值都包含前一条记录的哈希，形成不可篡改的链条。

hash(n) = SHA-256(data(n) + hash(n-1))

第六章：基础设施安全

6.1 托管环境

LabAnimal 的前端部署在 Vercel，这是一个全球 CDN 平台，内置 DDoS 防护和 Web 应用防火墙（WAF）。API 服务器部署在安全的云服务器上，使用防火墙和安全组限制访问。

6.2 网络安全

  • CORS 配置：严格的跨域资源共享策略，只允许白名单域名访问
  • 速率限制：API 端点实施速率限制，防止暴力攻击和 DDoS
  • 输入验证：所有用户输入使用 Zod schema 验证，防止注入攻击
  • SQL 注入防护：Prisma ORM 使用参数化查询，杜绝 SQL 注入

6.3 数据库安全

PostgreSQL 数据库使用以下安全措施：

  • SSL 加密连接
  • 行级安全策略（RLS）
  • 定期备份
  • 访问日志记录

第八章：安全运维实践

LabAnimal 采用以下安全运维实践：

  • CI/CD 安全扫描：每次代码提交都触发安全扫描，包括依赖审计和硬编码密钥检测
  • 依赖更新：定期更新依赖项，修补已知安全漏洞
  • 密钥管理：所有敏感配置使用环境变量，不硬编码在代码中
  • 日志监控：记录所有安全相关事件，便于审计和故障排查
  • 备份策略：定期数据库备份，支持灾难恢复

第九章：漏洞响应流程

LabAnimal 建立了完整的漏洞响应流程：

  1. 接收：通过 security@labanimal.tech 或 GitHub Security Advisories 接收漏洞报告
  2. 确认：48 小时内确认收到并开始评估
  3. 分类：按 CVSS 评分分类严重程度
  4. 修复：高危漏洞 7 天内修复，中危 30 天内修复
  5. 发布：发布安全补丁并通知用户
  6. 公开：修复后 90 天公开漏洞详情
