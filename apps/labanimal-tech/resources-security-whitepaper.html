Chapter 1: Security Overview & Design Philosophy

LabAnimal treats data security as its highest priority. As a laboratory animal compliance management system, we handle data involving research protocols, animal welfare records, and audit trails — the integrity and confidentiality of this data is critical.

Our security design follows these core principles:

  • Defense in Depth: Multiple layers of security controls, not relying on a single security mechanism
  • Least Privilege: Every component has only the minimum permissions needed to do its job
  • Zero Trust: All requests must be verified and authorized
  • Transparent Open Source: Core security implementation is publicly auditable

Chapter 2: Data Encryption Architecture

2.1 Transport Encryption

All data transfers are encrypted with TLS 1.3, the most secure version of the transport layer protocol available today. We enforce HTTPS, and all HTTP requests are automatically redirected to HTTPS.

Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

HSTS (HTTP Strict Transport Security) header ensures browsers always use HTTPS connections, preventing downgrade attacks.

2.2 Storage Encryption

Passwords are stored using bcrypt hashing with a cost factor of 12. This means that even if the database is compromised, attackers cannot crack passwords within a reasonable timeframe.

Password Hash Example:
Original Password: MySecurePass123
bcrypt Hash: $2b$12$LJ3m4ys3Lz0YBODCJn.MqOXPgFaHsVpJSKFTLKYqlVJTFHY.GGHKG

Chapter 3: Multi-Tenant Data Isolation

LabAnimal uses PostgreSQL Row Level Security (RLS) to implement strict multi-tenant data isolation. This is a database-level forced isolation — even if application code has a vulnerability, cross-tenant data access is impossible.

CREATE POLICY tenant_isolation ON animals
  USING (lab_id = current_setting('app.current_lab_id'));

Every database query automatically appends tenant filtering conditions, ensuring users can only access data from their own lab. This isolation is mandatory and cannot be bypassed.

Chapter 4: Authentication & Authorization

LabAnimal uses JWT (JSON Web Token) for stateless authentication. Tokens are signed with the RS256 algorithm, an asymmetric signing algorithm that is more secure than symmetric signing.

4.1 Password Policy

  • Minimum 8 characters
  • Must contain uppercase, lowercase letters and numbers
  • bcrypt encrypted storage (cost factor 12)
  • Login failure rate limit: 5 attempts/minute

4.2 Role Permissions

Role-Based Access Control (RBAC) defines four roles: Owner, Admin, Member, and Viewer, each with different permission sets.

Chapter 5: Audit Trail & Compliance

To meet 21 CFR Part 11 compliance requirements, LabAnimal implements a complete audit trail system. Each audit record includes the operation type, operator, timestamp, and data snapshot.

Audit logs use SHA-256 hash chain technology — each record's hash includes the previous record's hash, forming a tamper-proof chain.

hash(n) = SHA-256(data(n) + hash(n-1))

Chapter 6: Infrastructure Security

6.1 Hosting Environment

LabAnimal's frontend is deployed on Vercel, a global CDN platform with built-in DDoS protection and Web Application Firewall (WAF). The API server is deployed on secure cloud servers with firewalls and security groups restricting access.

6.2 Network Security

  • CORS Configuration: Strict Cross-Origin Resource Sharing policy, only allowing whitelisted domains
  • Rate Limiting: API endpoints enforce rate limits to prevent brute-force attacks and DDoS
  • Input Validation: All user input validated with Zod schemas to prevent injection attacks
  • SQL Injection Protection: Prisma ORM uses parameterized queries, eliminating SQL injection

6.3 Database Security

The PostgreSQL database employs the following security measures:

  • SSL encrypted connections
  • Row Level Security policies (RLS)
  • Regular backups
  • Access log recording

Chapter 8: Security Operations Practices

LabAnimal employs the following security operations practices:

  • CI/CD Security Scanning: Every code commit triggers security scans, including dependency audits and hardcoded secret detection
  • Dependency Updates: Regular dependency updates to patch known security vulnerabilities
  • Secret Management: All sensitive configuration uses environment variables, never hardcoded in source code
  • Log Monitoring: All security-related events are logged for audit and troubleshooting
  • Backup Strategy: Regular database backups with disaster recovery support

Chapter 9: Vulnerability Response Process

LabAnimal has established a complete vulnerability response process:

  1. Receive: Vulnerability reports received via security@labanimal.tech or GitHub Security Advisories
  2. Acknowledge: Acknowledge receipt and begin evaluation within 48 hours
  3. Classify: Classify severity by CVSS score
  4. Remediate: High-severity vulnerabilities fixed within 7 days, medium within 30 days
  5. Release: Publish security patch and notify users
  6. Disclose: Publicly disclose vulnerability details 90 days after fix
